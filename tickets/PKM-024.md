## Summary

Deliberately deferred from `docs/plans/010_meta_deck_library_plan.md`: the
1,500 downloaded replays collapse to just **137 unique exact 60-card lists**
(top 20 cover 82% of games; 91% of instances use a list seen ≥10 times —
every opponent is a competitor's fixed submission). Merging these into the
PKM-022 library would give exact level-1 identification of the *current*
opponent pool.

Deferred because today's replay pool is low-elo (where our submissions
start); the decks that decide final rank will look like netdecked human
tournament lists, and the identifier's level-2/3 fallback covers unmatched
bot decks meanwhile.

## Trigger to pick this up

- We climb the ladder and want exact reads on the actual opponent pool, or
- Pre-deadline refresh (final 1–2 weeks before Aug 16): re-download replays
  (`scripts/download_replays.py`), re-extract lists, merge.

## Work

- Extract unique lists + frequencies from `data/replays/raw/` (the
  fingerprint logic exists in `scripts/analyze_meta.py`; exact-list
  extraction was prototyped 2026-07-11)
- Map each onto a PKM-022 archetype by core similarity; unmatched clusters
  become their own "bot-brew" library entries with replay frequency as prior
- Library format is source-agnostic — this is additive, no redesign

## Resolution (2026-07-12, plan 011 Phase 1)

Un-parked and done ahead of the original trigger — plan
`docs/plans/011_deck_based_heuristics_plan.md` made the replay library the
foundation of the deck-based-heuristics direction, since the human-tournament
library alone identified almost nothing on the current pool (89% level-3,
wrong whenever it committed).

Built `scripts/extract_replay_decks.py` (re-runnable): extracts both players'
lists from every replay, dedupes (137 unique lists confirmed), clusters into
67 bot archetypes by Pokemon-set fingerprint, computes core/flex, and merges
into `data/meta_decks/library.json` under `source: "replays"` tags with
frequency-based priors (`--replay-weight 0.85`; human lists keep the rest,
re-weightable for a pre-deadline refresh). One design change vs. the sketch
above: unmatched-vs-matched mapping onto PKM-022 archetypes was skipped —
*every* bot cluster is its own entry, and Bayesian elimination sorts overlap
naturally.

**Gate result** (`scripts/eval_deck_identifier.py`, 1,500 replays / 3,000
perspectives, merged library — vs. the 2026-07-11 human-only baseline):

| Metric | Before | After |
|---|---|---|
| Level 1 (exact list) | 2%, **0% correct** | **40%, 100% correct** (1190/1190) |
| Level 2 (archetype core) | 10%, ~0% correct | 52%, 84% core_subset correct |
| Level 3 (no read) | 89% | 8% |
| Turn first at level 2 / 1 | — | mean 2.1 / 7.3 |

Caveat: this eval is in-sample (library built from the same replay pool it's
scored on) — which is also the deployment scenario, since opponents *are*
that pool. Staleness as competitors resubmit is handled by re-running the
script after each fresh `download_replays.py` batch.

Library artifact tests updated for the merged dual-source form
(`tests/test_meta_deck_library.py`, 6 passing). Note: re-running
`fetch_limitless_decks.py` writes a limitless-only library — re-run
`extract_replay_decks.py` afterwards to re-merge.
