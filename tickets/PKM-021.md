## Summary

`heuristics.py:prizes_remaining()` counts non-`None` entries in a player's
`prize` array as "still remaining":

```python
def prizes_remaining(player: dict) -> int:
    return sum(1 for p in (player.get("prize") or []) if p is not None)
```

Real captured `obs` data (`heuristic_loop/logs/**/*.json`) shows `prize`
entries are **always `None`**, for both players, whether or not that prize
has been taken — a prize's contents are hidden even from its own owner
until taken (matches real Pokemon TCG rules; see docs/CABT.md's comment
"None = taken", which is wrong). What actually signals a taken prize is the
array **shrinking** — e.g. `[None]*6` early game down to `[None]*3` after
three prizes are taken. Since every entry is always `None`, this function
always returns `0`, regardless of how many prizes are actually left.

Confirmed while building `src/pokemon/determinize.py` for
`docs/plans/009_native_search_plan.md` Phase 1 — see that doc's Phase 1 section
for the raw evidence (prize-array-length-over-time trace).

## Impact

`heuristics_dragapult.py:571` (`_boss_orders_wins_game`) consumes this
value:

```python
my_prizes = prizes_remaining(ctx.me)
...
if hp is not None and dmg >= hp and my_prizes - prize_value(t.get("id")) <= 0:
    return True
```

With `my_prizes` always `0`, the condition collapses to
`0 - prize_value(t) <= 0`, which is true for any positive `prize_value` —
i.e. this "does this KO actually end the game" check is currently vacuous
and fires on any lethal-damage KO, not just ones that actually bring our
prize count to 0. Net effect: `_boss_orders_wins_game` behaves the same as
"is this KO lethal", silently dropping the "and it actually wins the game"
half of its stated purpose.

## Fix

`prizes_remaining(player)` should return `len(player.get("prize") or [])`
(array length, not a `None`-filtered count) — the array already shrinks as
prizes are taken, so its length *is* the remaining count.

## Note

Double-check no other caller of `prizes_remaining()` relies on the current
(broken) always-zero behavior before changing it — `_boss_orders_wins_game`
is the only current consumer found via grep, but re-check at fix time.
