## Progress (2026-07-06) — first real iteration of the loop

Ran this process for the first time against `data/recent_log.txt` — a real
Kaggle replay of our **submitted** bot (confirmed identical to
`submission.tar.gz`'s bundled `heuristics.py`), playing the Psychic deck as
"NaqibL" and losing to a "Hop's Snorlax/Phantump" deck ("takerusQ").

### Systemic replay-format bugs found first (blocking reliable analysis)

Reading the raw Kaggle-format replay JSON by hand (and with the existing
`scripts/parse_replay.py`) turned out to be unreliable for two reasons,
both now fixed generically in the new `scripts/analyze_heuristic_logs.py`
(see below) rather than per-card:

1. **Off-by-one `selected` field.** The option actually chosen for the
   decision shown at frame `i` is not stored on `vis[i]["selected"]` — it's
   on `vis[i + 1]["selected"]`. Confirmed empirically: shifting by one
   recovers a valid, in-range selection for 182/183 decisions in one real
   game, vs. 151/183 reading the field unshifted (and those 151 "valid"
   ones are mostly coincidental overlap for small `maxCount=1` menus, not a
   real signal). Concrete example: a Ciphermaniac's Codebreaking search
   (`minCount=2, maxCount=2`, 46 options) showed `selected=[0]` on its own
   frame — a 1-card answer to a 2-card requirement — while the *next*
   frame's `selected=[14, 27]` was invalid for *that* frame's own 5-option
   Main menu but exactly matched two Metagross entries in the search
   frame's option list. **This affects every past reading of a Kaggle
   replay in this project, including `scripts/parse_replay.py`'s
   `selected`/`valid` handling and the training-label plan in
   `docs/001_training_pipeline.md`/PKM-008 — worth fixing there before any
   behavioral-cloning work starts, or BC will train on wrong labels.**
2. **Area-blind option labels.** `catalog.format_option` (and
   `scripts/parse_replay.py`'s Kaggle-format label helper) assume a card
   option refers to hand, mislabeling bench/active/deck-area options (the
   PKM-017 "known cosmetic issue"). `scripts/analyze_heuristic_logs.py`
   resolves any area, for either player, generically by card id.

### Concrete heuristic gap found and fixed

With those two fixed, the real trace showed: Slowking got **6 separate
energy attachments** and evolved twice over the game, but was **never once
the active Pokemon** after evolving — every attach landed on it while
benched. Meanwhile whichever backup attacker was active (Mega Kangaskhan
ex / Latias ex / Fezandipiti ex) sat at 1 energy or 0 energy the entire
game, nowhere near their actual attack costs (checked against
`deck/001_psychic_deck_reference.md` and the reverse-engineered attack
catalog: Kangaskhan/Latias/Fezandipiti all need 3 energy, Slowking's Seek
Inspiration needs 2). **Zero attacks fired in the entire 183-step game.**

Root cause, two compounding heuristic gaps:
- `attach_energy_to_slowking` unconditionally targeted Slowking regardless
  of board position, stranding energy on a benched, already-loaded
  Slowking instead of feeding whichever Pokemon was actually active.
- `switch_to_backup_attacker` unconditionally preferred the backup
  attackers on every switch decision — nothing ever switched Slowking back
  into active once it left, even after it had enough energy to attack.
  `setup_active_prefers_slowking_line` only covers the initial
  `SETUP_ACTIVE_POKEMON` context, not later switches.

**Fixed** (2026-07-06): `attach_energy_to_slowking` → `attach_energy_to_
attacker` (feeds Slowking to its threshold, then falls back to feeding
whichever Pokemon is active); `switch_to_backup_attacker` now prefers a
Slowking that's already loaded over the backup attackers. Both use a new,
data-driven `catalog.min_attack_energy_cost(card_id)` (reads real attack
costs from the reverse-engineered catalog — not hardcoded per Pokemon) so
this generalizes to any card, not just this deck's specific attackers.
Added/updated 6 unit tests in `tests/test_heuristics.py`.

**Also found while preparing to hand off the fix:** `submission.tar.gz`
never bundled `reverse-engineering/data/*.json`, so `catalog`'s data-backed
lookups (now including `min_attack_energy_cost`) would silently return
empty/`None` on Kaggle — the fix above would have partially no-opped in
production. Rebuilt the tar with those two JSON files included (1.2MB
total) and smoke-tested by extracting fresh and importing `main.agent`.

**Not yet done:** re-running a batch to confirm the fix actually improves
win rate (blocked on the user testing the new `submission.tar.gz` on
Kaggle themselves — local `.venv` is currently broken, `uv run` fails on a
Windows-symlink permission error removing `.venv/lib64`). See PKM-020 for
the "verify every heuristic change actually helps" harness this pointed
at.

### New tool: `scripts/analyze_heuristic_logs.py`

Built per this ticket's own suggestion ("should this become a script... once
the same log-reading steps get repeated a few times") — this session hit
exactly that trigger (huge raw JSON, the off-by-one bug, the area-blind
labeling bug, all rediscovered from scratch). Deck-agnostic: resolves any
card/attack for either player generically by id, nothing hardcoded to the
Psychic deck. Outputs a condensed per-turn decision trace plus an
end-of-game summary (option-type counts, attacks used, picks by select
context) instead of raw board-state JSON.

```bash
python scripts/analyze_heuristic_logs.py data/recent_log.txt --player 1
python scripts/analyze_heuristic_logs.py data/recent_log.txt --summary-only
```

## Summary

Use verbose game logs as the feedback signal for improving
`DEFAULT_PSYCHIC_HEURISTICS`, instead of hand-picking the next heuristic to
add/fix from intuition alone. An agent (or a script an agent drives) reads
a batch of `play -a heuristic -g N -v` traces, identifies concrete
recurring mistakes (a heuristic firing when it shouldn't, a decision point
where no heuristic fires and the random fallback picks something bad, a
resource spent on the wrong target), and proposes/implements a specific
heuristic change in response.

## Motivation

The Seek Inspiration bug fixed in PKM-017 (2026-07-05) is the model for
this: reading a live WSL playtest trace showed `prefer_copy_fodder_targets`
ranking Metagross/Kyurem ahead of Slowking/Slowpoke for *every* search
context, including `TO_HAND` — which meant Ultra Ball/Poke Pad kept
spending themselves pulling Seek Inspiration's discard fodder out of the
deck and into hand, where it was useless. That fix came from eyeballing
one verbose trace by hand. That doesn't scale as more heuristics get
added — want a repeatable process (loop, not one-off audit) for turning
"the agent lost, here's the log" into "here's the specific heuristic gap
and a proposed fix."

## Approach (subject to revision)

1. Run a batch of verbose games (`play -a heuristic -g N -v`) and save logs
   — probably to `data/` or a scratch dir, not committed.
2. Fix the known cosmetic bug first (`catalog.format_option` mislabels
   bench/active/deck-area options — see AGENTS.md "Known issues" and
   PKM-017): reading logs by eye to find errors is unreliable while option
   labels can be wrong. Low effort, high leverage for this ticket
   specifically.
3. For losses (and close/long games), identify:
   - Points where `fallback random` fired and the random pick was
     clearly bad (e.g. discarding a key piece, attaching energy to the
     wrong Pokemon, retreating into a worse attacker)
   - Points where an existing heuristic fired but produced a bad outcome
     (wrong priority ordering, a heuristic firing in a context it
     shouldn't, e.g. the PKM-017 bug)
   - Recurring patterns across multiple games, not one-off variance
4. Turn each finding into a proposed heuristic change (new rule, reordered
   priority, tightened context gate) with the specific log evidence cited
   — same evidence bar as the PKM-017 fix (quote the actual option/attack
   ids and context, not just "seemed suboptimal").
5. Implement, add/adjust unit tests in `tests/test_heuristics.py`, then
   re-run a batch and compare win-rate before/after (same n, same
   opponent) to confirm the change actually helped before keeping it.

## Open questions

- How many games are enough to trust a signal over noise? PKM-017's 20-game
  batches (0% -> 25%) are a reasonable current baseline for "believable
  swing," but worth deciding a minimum n before calling a change confirmed.
- Should this become a script (`scripts/analyze_heuristic_logs.py`) that
  an agent runs and summarizes, or stay a fully manual "run it, read it,
  fix it" loop for now? Leaning toward starting manual and only automating
  once the same log-reading steps get repeated a few times.
- Worth capturing *why* a heuristic fired (not just that it fired) in
  verbose output, to make this analysis less dependent on reconstructing
  intent from raw option dicts each time.
