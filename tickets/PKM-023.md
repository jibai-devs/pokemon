## Summary

Build `src/pokemon/deck_id.py`: a Bayesian-elimination belief over the
PKM-022 library, updated from the opponent's cumulative reveals (tracked in
`ctx.state`), degrading gracefully through three levels — exact list match →
archetype core match → true fringe (catalog-computed per-Pokémon threat
defaults). Phase 2 of `docs/plans/010_meta_deck_library_plan.md`.

## API (consumed by heuristics and, later, determinized search)

- `archetype_belief() -> dict[str, float]` — meta-share-weighted posterior
- `opp_remaining(card_id) -> (lo, hi, expected)` — exact count at level 1
- `p_in_hand(card_id) -> float | None` — hypergeometric over known remaining
  deck (levels 1–2)
- `identified_list() -> dict[int, int] | None`

## Deliverables

- Reveal tracker (visible zones + duplicates count as evidence) persisted in
  `ctx.state`
- Belief/elimination engine with meta-share prior; similarity weighting favors
  Pokémon evolution lines / draw engine over trainers
- Replace `archetype_latch`/`TIER5_SIGNATURES` in `heuristics_dragapult.py`
  with a soft belief read (keep the latch as fallback until parity is shown)
- **Offline eval harness**: drive the identifier with the 1,500 downloaded
  replays' reveal sequences; report by-turn belief concentration, level-1/2/3
  distribution, and accuracy vs the known submitted deck. Bot decks mostly
  won't exact-match tournament lists — this deliberately measures level-2/3
  fallback quality.
- Synthetic-obs pytest coverage; win-rate non-regression via
  `heuristic_loop/eval_heuristic_change.py`

## Downstream (not this ticket)

- Determinization prior for `SearchBegin` (plan 009 / PKM-015):
  `identified_list()` minus reveals ≈ exact hidden zones
- Per-archetype playbooks / posture dials keyed on `archetype_belief()`

## Progress (2026-07-11)

Built `src/pokemon/deck_id.py`: `DeckIdentifier` class with the full API
(`archetype_belief`, `opp_remaining`, `p_in_hand`, `identified_list`, plus
`level()`/`best_archetype()`/`archetypes()` helpers). Reveal tracking is the
running max of simultaneously-visible opponent card ids across
discard/hand/active/bench (incl. attachments/pre-evolutions), so movement
between zones (e.g. Night Stretcher) doesn't un-reveal a card. Elimination
is weighted: a Pokemon reveal beyond an archetype's core/flex cap
(weight 3.0) blows the consistency budget outright; a Trainer/Energy
overage (weight 1.0) is tolerated up to a small budget — approximates the
plan's "evolution lines / draw engines are most identity-revealing" via a
coarse Pokemon-vs-everything-else split (the catalog has no finer "engine"
tag). Level 1/2/3 degrade as designed; level 2 only fires when reveals have
*genuinely* narrowed the archetype set (not just meta-share dominance with
zero evidence).

Wired in non-invasively: `deck_belief_update` (side-effect heuristic, same
shape as `archetype_latch`) populates `ctx.state["deck_id"]` every decision
in `DRAGAPULT_HEURISTICS`, registered right after the existing latch.
**Did not** replace `archetype_latch`/`TIER5_SIGNATURES` — the library's
archetypes are the human tournament meta (Dragapult, Raging Bolt Ogerpon,
N's Zoroark, ...) and don't overlap `TIER5_SIGNATURES`' keys (low-elo bot
meta: arboliva, mega_lucario, crustle_kangaskhan, ...), so a real swap needs
the per-archetype playbook mapping the plan itself defers to Phase 3
("listed for orientation" only) plus a win-rate parity check. Doing that
half-baked here risked a real regression with nothing to catch it.

Offline eval harness: `scripts/eval_deck_identifier.py`, reports final
level distribution, turn-of-first-level-1/2, and accuracy vs the actual
submitted list, written to `data/meta_decks/deck_id_eval_report.txt`. Run
against the full 1,500-replay pool, both perspectives (3,000 identifier
runs): level 3 (fringe) 89%, level 2 (archetype core) 10%, level 1 (exact
list) 2%; belief first reaches level 2 by turn ~1.6 on average, level 1 by
turn ~2.8 when it gets there at all. Matches the plan's expectation —
today's bot meta mostly doesn't match tournament lists, so this is measuring
level-2/3 fallback quality, not a level-1 hit rate. Final-level accuracy vs.
the actual submitted list: exact_miss 2%, core_miss 10%, core_subset <1%,
n/a (level 3) 89% — the identifier is *confident* on 12% of games but wrong
more often than right when it commits at level 1/2 against this
out-of-domain replay pool, which is exactly the honest signal the harness
was built to surface (worth revisiting once/if PKM-024's replay-list merge
happens).

Tests: `tests/test_deck_id.py`, 12 cases against a synthetic library
(elimination, level transitions, exact/range remaining counts,
hypergeometric `p_in_hand`, the Pokemon-vs-Trainer weighting split) plus a
smoke test against the real `library.json`. Full suite: 71/71 passing.

Note: `deck_id.py`/`eval_deck_identifier.py` are pure Python (no engine
dependency), so this session's tests and eval runs were done via native
Windows Python rather than the usual WSL path — WSL's Hyper-V VM hit
`HCS/0x8007005a` ("insufficient system resources") mid-session and wouldn't
restart. Re-run via `uv run pytest` in WSL once it's available, as a
sanity check that nothing environment-specific slipped through.
