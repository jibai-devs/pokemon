## Summary

Whenever `heuristics.py` changes (by hand or by an agent), automatically run
a batch of games on the old version and the new version against the same
opponent, and report the win-rate delta — so "this heuristic change helped"
is a measured result, not an assumption, before the change gets kept.

## Motivation

PKM-019's own process already calls for this ("re-run a batch and compare
win-rate before/after... to confirm the change actually helped before
keeping it"), but so far it's been done manually and only sometimes (PKM-017
had a real before/after: 0/10 -> 5/20 after the Seek Inspiration fix). The
`attach_energy_to_attacker`/`switch_to_backup_attacker` fix from PKM-019's
2026-07-06 session has *not* been win-rate validated yet — it's reasoned
from a single game's trace, which is exactly the kind of change that's easy
to get subtly wrong (e.g. the new "feed the active attacker" fallback could
starve Slowking in some other line, or the switch-back could pull Slowking
into danger against a deck with sudden burst damage).

This matters more now that heuristic edits may come from an agent (reading
`scripts/analyze_heuristic_logs.py` output and proposing a fix) rather than
only a human who remembers to check — a harness makes "did this help"
mandatory and mechanical rather than a step that's easy to skip under time
pressure.

## Proposed shape (subject to revision)

- A script (e.g. `scripts/eval_heuristic_change.py`) that: checks out /
  imports two versions of `DEFAULT_PSYCHIC_HEURISTICS` (e.g. current
  working tree vs. last commit, or two explicit refs), runs N games of each
  against the same baseline opponent (`random_agent`, matching
  `pokemon.cli play`), and reports win/loss/draw counts + win-rate delta.
- Needs a minimum-n recommendation before calling a result "confirmed" —
  PKM-019 flagged this as an open question; PKM-017's 20-game batches
  (0% -> 25%) are the only real precedent so far. Worth a quick check of
  the variance in win-rate across repeated 20-game batches of the *same*
  heuristic set before trusting any single before/after comparison.
- Should work from a WSL/local run today (the only environment that can
  actually execute `libcg.so` games); Kaggle-side confirmation (real
  matchmaking results) is a separate, slower signal and not a substitute —
  see PKM-019's 2026-07-06 note about `submission.tar.gz` needing the
  bundled `reverse-engineering/data/*.json` to behave the same as local.
- Output should be simple enough for an agent to read and act on directly
  (e.g. "OLD: 4/20 (20%), NEW: 9/20 (45%), keep the change" or "NEW: 2/20
  (10%), revert") rather than requiring a human to eyeball two verbose logs.

## Depends on

PKM-017 (the heuristic agent itself) and PKM-019 (the log-reading process
this is meant to close the loop on — a heuristic change proposed from log
analysis should get validated by this harness before being considered done).
